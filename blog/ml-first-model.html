My First ML Model: What Worked, What Failed, and What I Learned

Building my first logistic regression model felt like magic — until it didn't. A candid breakdown of the mistakes, the debugging process, and the moment it finally clicked.

By Farhan

The first time I ran a logistic regression model and saw an accuracy score of 94%, I genuinely thought I was a machine learning engineer. I screenshot it. I was proud. I was also completely wrong about what that number meant.

This is the honest story of building my first real ML model — what I got wrong, what I had to unlearn, and what I'd tell myself if I could start over.

The Setup

I was working with a binary classification dataset — the goal was to predict whether a customer would churn or not based on behavioural features. Classic beginner project. I loaded the data, ran LogisticRegression().fit(X, y), called .score() and saw 94%. Done, right?

Not even close.

Mistake 1 — I Didn't Look at the Class Distribution

That 94% accuracy? The dataset was 93% class 0 (no churn) and only 7% class 1 (churn). My model had learned to predict "no churn" for almost everything and still scored 94%. It was useless for the actual problem — predicting who would churn.

This is called the accuracy paradox and it's one of the most common traps in beginner ML work. If I'd just plotted a value counts chart before touching the model, I would have caught it immediately.

What I did: Used class_weight='balanced' in scikit-learn and applied SMOTE to oversample the minority class. Accuracy dropped to 81% but my F1-score on the churn class went from 0.12 to 0.87. That's the number that actually matters.

Mistake 2 — I Trained and Tested on the Same Data

My first iteration didn't use a train-test split. I was evaluating my model on the exact data it had learned from — of course it performed well. This is data leakage and it makes your model completely unreliable in the real world.

What I did: Applied train_test_split with an 80/20 split and stratified sampling so both splits had the same class ratio. Suddenly the model's real performance was visible.

Mistake 3 — Raw Features Straight Into the Model

I had features on completely different scales — one column ranged from 0 to 1, another from 1000 to 900,000. Logistic regression is sensitive to feature scale because it uses gradient descent. My model was essentially ignoring the small-scale features.

What I did: Built a proper preprocessing pipeline using StandardScaler for numerical features and OneHotEncoder for categoricals, wrapped in a ColumnTransformer. This became the reusable pipeline I now use as a starting point for every model.

The Moment It Finally Clicked

After fixing all three issues, I ran the evaluation properly — confusion matrix, classification report, ROC-AUC curve. The model wasn't perfect. But it was honest. I could see exactly where it struggled, which classes it confused, and where more data or feature engineering would help.

That's the real skill in ML — not getting a high number, but understanding what your number actually means and knowing how to interrogate it.

What I'd Tell Myself on Day One

Before you touch a model, spend 80% of your time understanding your data. Plot everything. Check class distributions. Look at missing values. Understand what each feature actually represents in the real world.

A model is only as honest as the pipeline that feeds it. Get the pipeline right and the model almost takes care of itself.

The final model achieved 89% accuracy and an F1-score of 0.87 on the minority class — results I'm actually confident in because I understand exactly how they were produced. That confidence is worth more than any impressive-looking number.
